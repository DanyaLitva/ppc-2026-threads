## 1. Контекст

Отчет описывает гибридную реализацию алгоритма, объединяющую парадигмы распределенной (MPI - Message Passing Interface) и общей (OpenMP) 
памяти. Подобные архитектуры востребованы в современных вычислительных кластерах (HPC-системах).

Цель:
- исследовать модель межпроцессного взаимодействия (MPI) в сочетании с внутриядерной потоковой параллелизацией (OpenMP) для максимизации
 вычислительной плотности.

---

## 2. Постановка задачи

Обеспечить корректную работу алгоритма в изолированных адресных пространствах процессов, с необходимостью передачи, редукции и валидации
 данных через сетевой или локальный интерфейс обмена сообщениями. Главный (корневой) процесс должен выступать оркестратором операций 
 распределения и слияния.

---

## 3. Архитектура распределенных вычислений (уровень MPI)

Топология обмена сообщениями состоит из строго регламентированных коллективных операций:

1. **Инициализация (Broadcast)**: Корневой процесс (`rank 0`) транслирует размерность задачи (n) всем узлам коммуникатора посредством `MPI_Bcast`.
2. **Декомпозиция данных (Scatter)**: Формируются метаданные смещений (`displs`) и квот (`sendcounts`). Осуществляется рассылка сегментов данных рабочим процессам через вызов `MPI_Scatterv`.
3. **Локальная обработка**: Каждый узел в изолированном контексте инициирует локальную сортировку полученного буфера `local_vec` (гибридный вызов OpenMP).
4. **Агрегация (Gather)**: Отсортированные фрагменты возвращаются на корневой узел при помощи `MPI_Gatherv`.
5. **Финальная редукция**: Корневой процесс выполняет последовательное бинарное слияние полученных блоков (`std::inplace_merge`).
6. **Верификация**: В рамках интеграции с фреймворком тестирования итоговый результат транслируется всем процессам (`MPI_Bcast`) для успешного прохождения assert-проверок на каждой реплике.

---

## 4. Архитектура общей памяти (уровень OpenMP)

Внутри контекста локального процесса (узел кластера) полученный буфер `local_vec` дополнительно сегментируется для загрузки локальных 
процессорных ядер.
Для этого используется директива `#pragma omp parallel for`, функционально аналогичная той, что применялась в монолитной OMP-реализации.

---

## 5. Модели синхронизации

- **Уровень MPI**: Асинхронность нивелируется строгим соблюдением барьеров коллективных операций (`MPI_Scatterv`, `MPI_Gatherv`).
- **Уровень OpenMP**: Отсутствие гонок данных за счет независимых итерационных пространств циклов внутри процесса.

## Приложение

Листинг MPI-коммуникации с вложенным вызовом OpenMP-оркестрации:

```cpp
  MPI_Scatterv(world_rank == 0 ? vec.data() : nullptr, sendcounts.data(), displs.data(), MPI_INT, local_vec.data(),
               sendcounts[world_rank], MPI_INT, 0, MPI_COMM_WORLD);

  ShellSortMerge(local_vec); // Вызов гибридного OpenMP ядра

  MPI_Gatherv(local_vec.data(), sendcounts[world_rank], MPI_INT, world_rank == 0 ? vec.data() : nullptr,
              sendcounts.data(), displs.data(), MPI_INT, 0, MPI_COMM_WORLD);

  if (world_rank == 0) {
    for (int i = 1; i < world_size; ++i) {
      if (sendcounts[i] > 0) {
        std::inplace_merge(
            vec.begin(), vec.begin() + static_cast<std::ptrdiff_t>(displs[i]),
            vec.begin() + static_cast<std::ptrdiff_t>(displs[i]) + static_cast<std::ptrdiff_t>(sendcounts[i]));
      }
    }
  }
```
