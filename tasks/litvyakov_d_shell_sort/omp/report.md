## 1. Контекст

Рассматривается параллельная реализация алгоритма сортировки Шелла с простым слиянием с использованием стандарта OpenMP.

Цель:
- уменьшить время выполнения сортировки по сравнению с SEQ-версией
- исследовать масштабируемость алгоритма на многоядерных системах

---

## 2. Постановка задачи

Необходимо отсортировать входной массив (вектор) целых чисел по возрастанию с применением многопоточности.

### Условия корректности
- Результирующий массив должен быть упорядочен по неубыванию.
- Алгоритм должен корректно обрабатывать массивы любых размеров (включая не кратные количеству потоков).
- Сравнение с последовательной реализацией (SEQ) на идентичных тестовых данных.

---

## 3. Базовый алгоритм

Используется многопоточная адаптация двухфазного алгоритма:

1. **Фаза сортировки**: 
   - Массив делится на количество частей, равное количеству доступных потоков (или размеру массива, если он меньше числа потоков).
   - Вычисляются точные границы каждого блока памяти (используется функция `GetBounds`, распределяющая остаток деления равномерно).
   - Каждый подмассив сортируется базовой сортировкой Шелла.

2. **Фаза слияния**: 
   - Выполняется последовательное слияние отсортированных блоков in-place (с использованием `std::inplace_merge`).

Параллелизм вводится исключительно на первой фазе — независимой сортировке выделенных фрагментов массива.

---

## 4. Схема распараллеливания

### Область параллелизма
Параллелизуется цикл обработки подмассивов:
- каждый поток берет на себя одну или несколько итераций цикла по количеству частей массива (`parts_count_t`).
- используется директива `#pragma omp parallel for schedule(static)`.

### Разделение данных
- `vec`, `bounds`, `parts_count_t` — **shared** (доступны всем потокам для чтения/записи в непересекающиеся области памяти).
- Внутренние индексы `l` и `r` — **private** (локальные для потоков).

### Синхронизация
- Отсутствует внутри фазы сортировки (каждый поток работает со своим уникальным диапазоном вектора `[l, r)`).
- Отсутствуют `critical`, `atomic`.

### Барьеры
- Неявный барьер в конце директивы `#pragma omp parallel for` перед переходом к фазе последовательного слияния.

## 5. Детали реализации

Файлы:
- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Pipeline реализации:
- **ValidationImpl**: проверка на пустоту.
- **PreProcessingImpl**: копирование входных данных.
- **RunImpl**:
  - вычисление границ частей массива.
  - параллельный вызов `BaseShellSort` на заданных диапазонах.
  - последовательный вызов `std::inplace_merge` для объединения отсортированных блоков.
- **PostProcessingImpl**: завершение работы.

### Особенности
Использование `std::inplace_merge` позволяет избежать лишних аллокаций памяти (по сравнению с ручным выделением `left` и `right` в версии SEQ). Однако фаза слияния остается строго последовательной, что накладывает ограничения на максимальное ускорение (согласно закону Амдала).

Наблюдения:
- Уверенный рост производительности наблюдается вплоть до 8 потоков. Максимальное ускорение достигает 3.75 раз.
- С увеличением числа потоков эффективность начинает предсказуемо снижаться.
- Падение эффективности обусловлено последовательной фазой слияния (цикл с `std::inplace_merge`), которая начинает занимать всё больший процент от общего времени по закону Амдала.

## Приложение

Фрагмент показывает распределение данных по потокам в OMP. 

#pragma omp parallel for default(none) shared(vec, bounds, parts_count_t) schedule(static)
  for (int i = 0; i < parts_count_t; ++i) {
    const std::size_t l = bounds[i];
    const std::size_t r = bounds[i + 1];
    BaseShellSort(vec.begin() + static_cast<std::ptrdiff_t>(l), vec.begin() + static_cast<std::ptrdiff_t>(r));
  }

  for (std::size_t i = 1; i < parts_count; ++i) {
    std::inplace_merge(vec.begin(), vec.begin() + static_cast<std::ptrdiff_t>(bounds[i]),
                       vec.begin() + static_cast<std::ptrdiff_t>(bounds[i + 1]));
  }
