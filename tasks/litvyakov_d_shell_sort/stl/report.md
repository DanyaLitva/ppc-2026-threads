## 1. Контекст

Рассматривается параллельная реализация с использованием стандартной библиотеки потоков C++ (`std::thread`). Данная реализация выполняется в качестве проверки способностей ручного распределения вычислительных задач потокам без использования сторонних фреймворков и библиотек.

Цель:
- реализовать параллельную версию при помощи `std::thread`, используя ручной параллелизм по паттерну `create -> work -> join`.

---

## 2. Постановка задачи

Необходимо отсортировать массив целых чисел по возрастанию с обеспечением корректной обработки данных в многопоточной среде.

### Условия корректности
- Результирующий массив должен быть отсортирован по неубыванию.
- Отсутствие гонок данных при доступе к массиву.
- Совпадение результатов с последовательной (SEQ) версией алгоритма.

---

## 3. Базовый алгоритм

Алгоритм полностью повторяет логику версий OMP и TBB:
1. Вычисление количества рабочих кусков `parts_count`.
2. Определение границ кусков вектора с учетом остатка от деления.
3. Инициализация потоков, каждый из которых сортирует свой подмассив.
4. Последовательное слияние готовых подмассивов.

---

## 4. Схема распараллеливания

### Область параллелизма
Параллелизм достигается ручным созданием пула потоков `std::vector<std::thread> thread_pool(parts_count)`.

1. Каждому потоку через лямбда-функцию передаются точные индексы `l` и `r`.
2. Поток выполняет функцию `BaseShellSort` над диапазоном `[vec.begin() + l, vec.begin() + r)`.
3. Главный поток ожидает завершения работы всех порожденных потоков в цикле при помощи `.join()`.

### Разделение данных
- `vec` передается по ссылке, но каждый поток оперирует строго непересекающимися отрезками памяти.
- Индексы `l` и `r` копируются (захватываются по значению) в лямбду каждого потока.

### Синхронизация и Барьеры
- Явный барьер синхронизации реализован через `t.join()` для всех элементов `thread_pool`. 
- Не используются мьютексы, так как области памяти изолированы.

## 5. Детали реализации

Файлы:
- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

Особенности реализации заключаются в отсутствии скрытых оптимизаций планировщика, характерных для OMP и TBB. Мы явно выделяем `parts_count` потоков, а системный планировщик ОС самостоятельно распределяет их по физическим ядрам.

## Приложение

Фрагмент показывает распределение данных по потокам в STL. 

  std::vector<std::thread> thread_pool(parts_count);

  for (std::size_t i = 0; i < parts_count; ++i) {
    const std::size_t l = bounds[i];
    const std::size_t r = bounds[i + 1];
    thread_pool[i] = std::thread([&vec, l, r]() {
      BaseShellSort(vec.begin() + static_cast<std::ptrdiff_t>(l), vec.begin() + static_cast<std::ptrdiff_t>(r));
    });
  }

  for (auto &t : thread_pool) {
    t.join();
  }
